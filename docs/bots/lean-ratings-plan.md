# Lean pilot/deck ratings — the 80/20 brief

**Status:** agreed direction, not yet executed. This is a *what & why* brief — the "how"
(exact commands, scoping, fitting details) is left to the implementing agent.

## Goal

Get a **rough absolute-strength number** for individual bots (pilots) and/or decks — a
single ladder you can read top-to-bottom. This is a *navigation aid*, not a publication.
We explicitly do **not** need rigorous data science here.

## What we're doing

Fit an anchored Bradley–Terry-style rating (RandomBot pinned at 0) to get **point
estimates only** — one strength number per pilot and per deck. Decompose into pilot vs
deck effects so a result tells us *which lever to pull*, consistent with the
never-nerf-a-card-to-fix-a-bot triage.

## What we're deliberately cutting (and why)

- **No confidence intervals / bootstrap.** For a rough ladder we don't need error bars.
- **No first-player / seat term.** Going first vs second is a real effect but a
  far-future refinement; fold it away for now.

The important insight behind these cuts: **they save almost nothing, because the cost is
not the statistics.** The fit (and even the bootstrap) is seconds of compute. The
expensive part is *playing the games*. So the real lever for effort is **fewer games** —
fewer/cheaper pilots — not trimming the math. Cutting CIs and seat is fine; just don't
expect them to be where the time goes.

## The two goals are not equally cheap

- **Deck strength is cheap.** A deck's average win rate across the field is already a
  serviceable strength number, obtainable from a fixed-pilot matrix. A rating fit just puts
  it on a nicer additive scale.
- **Bot strength requires cross-play.** You cannot rank pilots from games where the same
  pilot sits in both seats — they must *face each other*. That means playing new games;
  there is no shortcut around it.

## Why we can't just reuse existing sim output

- Nothing reusable is saved: the report/sim pipeline aggregates straight to win-rate
  matrices and discards per-game outcomes; its records don't even carry pilot identity.
  The paired benchmark writes only aggregated CSVs. `results/` holds only sparse human logs.
- The default deck matrix is same-pilot (both seats one bot), so it can rate *decks* but
  says nothing about *pilot-vs-pilot* strength.
- A pilot-labelled cross-play win-rate table *would* be a valid input for the point
  estimates — aggregate counts are mathematically sufficient once seat and CIs are dropped —
  but that table doesn't exist on disk. It has to be generated by a cross-play run.

## The shape of the lean run

One **scoped cross-play cohort** over a small pilot set that **excludes the search-heavy
pilot** (the throughput killer), point estimates only. The existing ratings harness already
does cross-play and can be scoped; the implementing agent should choose the exact pilots,
sample size, and invocation.

## Open / unresolved

- **Runtime is unmeasured.** Cohort size scales with the number of pilot-pairs; dropping the
  slow search pilot cuts per-game time substantially. Measure throughput before committing to
  a full run.
- Keep the standing invariants: bots stay generalist, never read hidden info, and any card
  signal that emerges goes to Balance, not a bot patch.
